## Findings considered and rejected (this run, 2026-07-19)

- **Stripe webhook idempotency race** (`apps/api/app/webhooks/payments/route.ts:193`):
  the check-then-record window is real, but the subscription mirror is an upsert
  with a monotonicity guard and `recount-usage` is itself an upsert, so the
  blast radius today is duplicated work, not wrong data. Worth documenting the
  "all handlers must stay idempotent" invariant with a test; not worth
  restructuring the route.
- **Xero token exchange inside the advisory-lock transaction**
  (`packages/xero/src/oauth/service.ts:427-449`): holds a connection across
  network I/O, but the lock is precisely what prevents two processes consuming
  the same single-use refresh token, and `exchangeSucceeded` handles the
  commit-failure case. Deliberate and sound.
- **Feed-token expiry write scoped by id alone**
  (`packages/feeds/src/render/render-feed.ts:100`): the row was loaded by token
  hash so the ids are self-consistent; not exploitable. Recorded only as drift
  from the rule the same file documents twice at `:166` and `:144`.
- **Broadcast of `sync.run_status_changed` to all org members**
  (`packages/notifications/src/sse/broker.ts:99`): reaches viewers, but the
  payload is only run id, type and status.
- **`organisationId` taken from the request body** in
  `apps/api/app/api/availability/[recordId]/route.ts:65` and siblings: validated
  against `clerkOrgId` before use, and the services filter on both ids. Also
  rejected by the previous run; agreed.
- **Per-employee `sleep` in the Xero balance read**
  (`packages/xero/src/au/read.ts:234`): redundant against the limiter, which
  `xero-fetch.ts` describes as the "single choke point for every Xero call".
  Downgraded rather than rejected — a comment there notes the balance loop
  deliberately owns its retry semantics, so removing the sleep is a considered
  change, not an obvious cleanup.
- **Xero vocabulary in the canonical availability layer**: `CLAUDE.md:374` says
  Xero types must not leak into `packages/availability`, and at the type level
  they do not (zero `@repo/xero` imports). But `"xero_sync_failed"` is a
  first-class approval status and `xero_write_error` is a column. Renaming is an
  L-effort, HIGH-risk migration across a persisted enum. The right move is to
  decide and record it — either rename to connector-neutral values or amend
  `CLAUDE.md` to say the boundary is type-level only — not to refactor.
